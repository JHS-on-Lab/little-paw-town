# Little Paw Town - MVP Scope v0.1

---

## 1. 문서 목적

본 문서는 `Little Paw Town GDD v0.1`를 기준으로, **초기 출시 버전에서 실제로 구현할 범위와 제외할 범위**를 명확히 정의하기 위한 문서다.

핵심 목적은 다음과 같다.
- 감성형 반려동물 육성 게임의 **핵심 재미 검증**
- 종 확장 가능 구조를 유지하면서도 **초기 제작 복잡도 통제**
- 애정도/이벤트/추억 중심의 루프가 실제로 리텐션을 만드는지 확인

---

## 2. MVP 목표

MVP에서 검증할 질문은 다음 4가지다.

1. 유저는 **내 반려동물을 직접 만드는 과정**에서 애착을 느끼는가
2. 유저는 **오늘의 장면을 확인하러 다시 접속**하는가
3. 유저는 **추억과 애정도 누적**을 장기 동기로 느끼는가
4. 종 확장을 고려한 구조가 초기 개발 효율을 과도하게 해치지 않는가

---

## 3. MVP 설계 원칙

### 3.1 핵심 원칙
- MVP는 **콘텐츠 폭**보다 **감정 루프 완성도**를 우선한다.
- 유저가 매 세션마다 최소 1회는 **“오늘의 장면”**을 경험해야 한다.
- 애정도는 숫자 파밍이 아니라 **이벤트와 교감 결과**로 느껴져야 한다.
- 종 확장은 데이터 구조로 먼저 대비하되, 초기 플레이 가능 종 수는 제한한다.
- FOMO, 경쟁, 압박형 숙제 구조는 MVP 단계에서 배제한다.

### 3.2 구현 우선순위 원칙
- 1순위: 생성 → 홈 → 장소 이동 → 이벤트 → 추억/애정도 저장
- 2순위: 하우징, 간단 경제, 미니게임
- 3순위: 푸시형 돌발 이벤트, 세부 연출 고도화

---

## 4. 출시 기준 제품 정의

### 4.1 제품 한 줄 정의
현실 시간 분위기에 따라 달라지는 타운에서, 내가 만든 반려동물의 일상을 관찰하고 교감하며 **추억과 애정도**를 쌓는 감성 생활형 육성 게임.

### 4.2 MVP 플레이 감각
- 하루 1~2회 짧게 접속해도 장면이 남아야 함
- 직접 개입은 적지만, **선택의 의미**는 있어야 함
- 세션 종료 시 최소 하나는 남아야 함
  - 추억 카드
  - 애정도 상승
  - 관계 진전
  - 홈 변화

## 5. MVP 포함 범위

## 5.1 계정/진입
- 앱 실행
- 타이틀 화면
- 게스트 또는 간단 계정 진입 구조
- 튜토리얼 시작 플로우

> 판단: 계정 시스템은 최소화하고, 이탈 없이 바로 생성/플레이 진입이 가능해야 한다.

### 5.2 반려동물 생성
- 직접 생성 방식만 지원
- 지원 종은 **최소 1종 우선 구현**
- 구조적으로는 강아지/고양이 확장 대응
- 입력 항목
  - 이름
  - 종 선택 또는 종 슬롯 구조 확보
  - 기본 파츠 조합
  - 기본 색상/무늬/소품
  - 5개 공통 특성 초기 설정 또는 간접 결정

### 5.3 코어 플레이 루프
- 홈 진입
- 오늘의 분위기 표시
  - 시간대 4단계
  - 날씨 2종(맑음/비)
- 반려동물 상태/반응 확인
- 장소 이동
- 이벤트 발생
- 유저 반응 선택 또는 짧은 교감 행동
- 결과 저장
  - 추억
  - 애정도
  - 관계
  - 재화 또는 소품

### 5.4 장소
MVP 장소는 4곳으로 제한한다.
- 홈
- 공원
- 베이커리
- 광장

장소별 목표:
- 홈: 상태 확인, 상호작용, 꾸미기, 복귀 거점
- 공원: 기본 산책/놀이 이벤트
- 베이커리: 냄새/간식/생활형 해프닝
- 광장: 주민/친구 만남, 관계 이벤트 중심

### 5.5 이벤트 시스템
- 원형 이벤트 40~50개
- 구조: 원형 이벤트 + 조건 분기 + 후속 결과
- 분기 기준
  - 시간
  - 날씨
  - 장소
  - 특성 1~2개
  - 종 레이어 대응 가능 구조
  - 애정도 단계
- 이벤트 카테고리
  - 환경 반응
  - 습관 반응
  - 관계 이벤트
  - 애정도 이벤트
  - 연쇄 이벤트

### 5.6 애정도 시스템
- 플레이어 ↔ 반려동물 유대 수치
- 5단계 구조
  - 낯섦
  - 익숙함
  - 편안함
  - 신뢰
  - 깊은 애착
- 상승 원천
  - 이벤트 완료
  - 반응 선택
  - 특별 순간
  - 소규모 일상 상호작용
- 보상 방향
  - 홈 행동 변화
  - 선제 반응 증가
  - 전용 이벤트 해금
  - 특별 추억/장식 해금

> 판단: MVP에서는 애정도 하락을 넣지 않는 편이 적절하다. 초기 버전에서 하락은 죄책감과 이탈 리스크가 더 크다.

### 5.7 추억 시스템
- 이벤트 결과를 추억 카드 형태로 저장
- 카드 열람 가능
- 최소 메타데이터 저장
  - 이벤트 종류
  - 장소
  - 시간/날씨
  - 주요 문구 또는 컷
  - 획득 일시
- 특정 세트 수집 보상은 단순 해금 수준으로 제한

### 5.8 관계 시스템
- 주민 2명
- 동물 친구 1마리
- 친밀도 또는 관계 단계 저장
- 특정 관계 단계에서 전용 이벤트 해금

### 5.9 하우징
- 홈 소품 8~12종
- 배치/이동/저장
- 일부 소품은 반려동물 행동 반응 변화 유발

### 5.10 경제 시스템
- 기본 머니 1종
- 사용처
  - 장난감
  - 간식
  - 기본 하우징 소품
- 감성 토큰은 구조만 고려하고 MVP 정식 도입 제외

### 5.11 미니게임
- 1~2종
- 후보
  - 공놀이
  - 숨은 장난감 찾기
  - 냄새 추적
- 목적
  - 보조 재미
  - 소액 머니 보상
  - 애정도 보조 상승

### 5.12 돌발 이벤트 / 푸시
- 3~5종 템플릿
- 푸시 빈도 제한
- 동일 유형 쿨다운
- 손실 회피형 메시지 금지

### 5.13 기본 설정
- 사운드 on/off
- 푸시 on/off
- 계정/저장 정보 확인
- 언어/약관/문의 링크는 최소 범위로 준비

## 6. MVP 미포함 범위

다음 항목은 구조 고려만 하고, 초기 버전 구현에서는 제외한다.

### 6.1 콘텐츠/시스템 제외
- AI 반려동물 생성
- 계절 시스템
- 감성 토큰 고도화
- 포토부스
- 호수
- 골목 확장 탐험 루프
- GPS 기반 소셜
- 앨범 공유
- 제한적 소셜 기능
- 다두/다묘/복수 반려동물 동시 육성
- 기타 반려동물 본격 추가

### 6.2 운영 고도화 제외
- 실시간 시즌 이벤트 운영 툴
- 원격 밸런스 툴 정교화

- 복잡한 재화 2중/3중 구조

### 6.3 연출 고도화 제외
- 풀스크린 시네마틱 컷신 다수
- 복잡한 포즈/카메라 편집 기능
- 고급 포토 모드

## 7. 권장 초기 플레이 가능 종 판단

### 7.1 권장안
- **플레이 가능 종은 1종으로 시작**
- 데이터/이벤트/애니메이션 구조는 강아지/고양이 확장을 전제로 설계

### 7.2 이유
- 종이 늘어날수록 파츠, 애니메이션, 이벤트 분기, QA 비용이 기하급수적으로 증가함
- MVP의 핵심 검증 포인트는 종 다양성보다 **애정 루프와 재접속 동기**임
- 초기 종 1개로도 생성 애착, 이벤트 감상, 추억 누적, 하우징 루프는 충분히 검증 가능함

### 7.3 확장 조건
다음 조건을 만족하면 1차 업데이트에서 고양이 추가를 검토한다.
- 생성 완료율 충분
- D7 리텐션 기준 충족
- 홈/이벤트 루프의 만족도 확보
- 공통 이벤트 구조 재사용성이 확인됨

## 10. 세션 기준 경험 정의

### 10.1 짧은 세션 (1~3분)
- 홈 진입
- 반려동물 반응 확인
- 짧은 교감 1회
- 이벤트 1회
- 추억 또는 애정도 반영 후 종료

### 10.2 일반 세션 (5~10분)
- 장소 1~2곳 이동
- 이벤트 2~3회
- 관계 또는 애정도 진전 확인
- 미니게임 1회 또는 하우징 1회
- 결과 저장 후 종료

### 10.3 실패 기준
다음 상태면 MVP 루프가 약한 것으로 판단한다.
- 접속 후 유저가 해야 할 일이 모호함
- 이벤트가 단순 텍스트 소비로 끝남
- 애정도 상승이 체감되지 않음
- 추억 저장이 감정적 보상으로 작동하지 않음

## 11. 수치 및 밸런스 초안

### 11.1 애정도 초안 원칙
- 핵심 상승은 이벤트에서 발생
- 반복 상호작용은 소량 상승 또는 일일 효율 감소
- 하루 내 과도한 파밍 방지

### 11.2 예시 구조
- 일반 이벤트 완료: +3 ~ +8
- 애정도 이벤트 완료: +8 ~ +15
- 특별 첫 경험: +10 ~ +20
- 단순 상호작용: +1 내외

> 위 수치는 설계 검토용 초안이며, 구현 후 플레이 테스트 기준으로 재조정 필요.

### 11.3 머니 초안 원칙
- 생존 비용처럼 느껴지면 실패
- 일상 상호작용에 필요한 소비는 낮게 유지
- 장식/표현용 소비 비중을 높임

## 13. 개발 리스크

### 13.1 구조 리스크
- 종 확장 고려가 과하면 MVP 구조가 불필요하게 무거워질 수 있음
- 이벤트 분기 구조가 복잡해지면 제작/QA 비용이 급증함

### 13.2 콘텐츠 리스크
- 이벤트 수만 채우고 장면 밀도가 낮으면 감정 루프가 약해짐
- 애정도 보상이 숫자만 보이면 핵심 차별점이 사라짐

### 13.3 일정 리스크
- 생성 파츠 수가 늘어나면 UI/아트/저장 구조 비용이 급증함
- 하우징을 자유 배치형으로 깊게 가면 예상보다 공수가 커질 수 있음

## 15. 최종 정리

이 MVP는 **반려동물 종 다양성**을 먼저 보여주는 버전이 아니라, **애정도와 추억이 실제 애착을 만드는지 검증하는 버전**이다.

따라서 초기 범위에서는 다음을 반드시 지켜야 한다.
- 플레이 가능 종 수를 제한한다.
- 이벤트와 애정도 체감 품질을 우선한다.
- 추억 저장과 홈 변화가 실제 보상처럼 느껴지게 만든다.
- 구조는 확장 가능하게 설계하되, 구현 범위는 작게 유지한다.
